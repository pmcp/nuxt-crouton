Three Block Systems

Crouton has three distinct "block" concepts. They serve different purposes at different levels of the stack.

Composition Atelier Blocks

App-level capabilities on a kanban canvas

PurposeCompose which features the app has
Examplesschedule, hero, my-bookings, signup
Defined incrouton-atelier/data/blocks.ts
Registered viaStatic data (manifest Phase B)
Accessed viauseBlockRegistry()
Created byUser in Atelier UI
OutputAppComposition (Yjs Y.Map)

Content Editor Blocks

TipTap nodes in the page content editor

PurposeCompose what a page contains
ExamplesheroBlock, chartBlock, collectionBlock
Defined incrouton-core/types/block-def.ts
Registered viaapp.config.ts → croutonBlocks
Accessed viauseCroutonBlocks()
Created byUser in page editor (or Editor AI)
OutputTipTap JSON in page content

Routing Page Types

CMS page components in the page tree

PurposeRoute a page to its renderer
Examplesregular, booking, collection-detail
Defined incrouton-core/types/app.ts
Registered viaapp.config.ts → croutonApps.pageTypes
Accessed viausePageTypes()
Created byAdmin in page tree
OutputPage record in DB with pageType

AI Generation Pipeline

Four AI roles + CLI, sequential. Each step's output feeds the next. The result is a generated layer indistinguishable from a real package.

🏗

Architect

AI ROLE 1 — DATA MODEL

Talks to the user, understands the domain, designs the data model. Has full project context: what collections exist, what packages are installed, what relationships are in play.

Schemas
events.json, rooms.json, memberships.json, registrations.json
Seed data
seed/events.json, seed/rooms.json
Project context
collections, packages, relationships
🎨

Designer

AI ROLE 2 — COMPONENTS

Reads schemas. Decides how each collection should present itself. Adds hints for standard output or writes custom Vue components when the domain demands it.

Hints
rooms: $list=grid, $card=image-top
memberships: $list=table
Custom components
Events/List.vue (calendar)
Events/Card.vue (rich)
Events/Detail.vue
Events/Form.vue
Embedded views
Registrations/Form.vue
Registrations/List.vue
⚙

CLI

CODE GENERATION — INFRASTRUCTURE

Reads schemas with hints. Generates everything the Designer didn't touch: composables, types, API routes, database schema, standard components.

Composables
useEventsEvents()
useEventsRooms()
useEventsMemberships()
Server
API routes (CRUD)
Database schema (Drizzle)
Migrations
Standard components
Rooms/List.vue (hint: grid)
Rooms/Card.vue (hint: image-top)
Memberships/List.vue (hint: table)
+ all Form.vue, Detail.vue
Types
TypeScript definitions
Zod schemas
Collection metadata

Analyst

AI ROLE 3 — DATA INSIGHT

Bridges domain data with visualization packages. Reads collection schemas + available addons (charts, maps). Creates meaningful, pre-configured visualizations. Only runs when visualization packages are present.

Chart presets (if crouton-charts)
Booking Trends (monthly count)
Revenue by Location (grouped bar)
Popular Timeslots (heatmap)
Membership Growth (area)
Active Members (donut)
Map configs (if crouton-maps)
All Locations (collection map)

generated layer's app.config.ts

Editor

AI ROLE 4 — PAGE LAYOUT

Composes pages from all available pieces: built-in editor blocks, Designer components, CLI standard components, and Analyst visualizations. Arranges them into page layouts per visibility level.

Public landing
heroBlock → collectionBlock (events, calendar) → collectionBlock (rooms, grid) → collectionMapBlock (locations) → ctaBlock (signup)
Member dashboard
collectionBlock (my-bookings, list) → collectionBlock (events, card-row)
Admin overview
chartBlock (Booking Trends) ← Analyst
chartBlock (Revenue by Location) ← Analyst
collectionBlock (bookings, table)
collectionBlock (members, table)

Registration & Discovery Flow

How capabilities travel from package source to runtime availability.

1. Package Source

Author time
crouton.manifest.tsDeclares capabilities: editorBlocks, collections, composables, components, fieldTypes
app/app.config.tsRegisters runtime definitions: croutonApps, croutonBlocks (full CroutonBlockDefinition)
nuxt.config.tsLayer dependencies, component dirs, auto-imports
↓

2. Nuxt Build

Build time — deep merge
Layer mergeAll nuxt.config.ts merged into one config
App config mergeAll app.config.ts deep-merged: croutonApps + croutonBlocks combined
Auto-import scanComponents, composables discovered from all layers
↓

3. Runtime Composables

Runtime — useAppConfig()
useCroutonApps()hasApp(), adminRoutes, dashboardRoutes, pageTypes
useCroutonBlocks()hasBlock(), getBlock(), blocksList
usePageTypes()Aggregates page types from all apps
useCollections()Collection metadata from generated layers
↓

4. UI Consumers

Runtime — components read composables
Page editorDiscovers editor blocks → creates TipTap extensions dynamically → / menu
Sidebar navigationReads adminRoutes + dashboardRoutes from croutonApps
Page treeShows available page types from all packages
Public rendererLooks up pageType → renders component. Reads TipTap JSON → renders blocks.
Atelier canvasReads block registry → kanban with blocks → scaffold
