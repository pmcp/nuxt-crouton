The renderer owns its pane handles

Today a consumer paints a separate overlay layer of grab-faces over a composed card and keeps it aligned by measuring the rendered panes ([data-panel]) and clamping the rects every frame — the v49/v51 measure-and-clamp machinery exists only because the faces are a duplicated layer that drifts. This proposal moves the wiggle / reorder / detach affordances onto the rendered panes themselves: the renderer draws each handle inside its own pane and tags the pane with a stable id. No second layer, nothing to measure, the drift bug class is gone.

What changes

  • The renderer gains an editable mode — each rendered pane carries its own grab-handle (wiggle / reorder / detach) and a stable data-pane-id; the consumer no longer paints a separate overlay layer.
  • The overlay-drift bug class is gone — handles ride their panes through every reflow / stack / min-width clamp / responsive resize, so the v49/v51 measure-and-clamp machinery is deleted, not ported.
  • Detach uses a throwaway floating proxy only during the pull (the one case a pane must leave its slot); reorder stays in-place.
  • The five data-handoff hooks are reproduced verbatim — snap-guide[data-armed], ghost-pane, page-badge, floor-readout[data-hard-floor/data-soft-floor], region-pill[data-region] — so one exploratory agent runs identically on the POC and the graduated app.
  • Look is the POC's proven look (the running v52 is the visual ground truth) — this is an architecture move, not a redesign. Resting panes stay node-draggable; the green primary stays the snap/reorder signal.
