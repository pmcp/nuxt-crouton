How to read it

  • Humans pick their tool. CEO/PM in Slack & Linear, devs in GitHub & Linear, marketing in HubSpot, knowledge in Notion. Nobody is forced into one place.
  • GitHub is the engine. Whatever a person does in their tool, it lands as a GitHub issue so the agents can act on it. Linear mirrors it two-way for planning.
  • Agents are teammates. They pick up issues, fan work out to the right runner (cloud for web + screenshots, Mac mini for Apple), and post results back.
  • Three self-iterating loops hang off the engine: the agent loop (issue → PR → postmortem → better issue), the design-system loop (Storybook ↔ Figma ↔ code), and the docs loop (code ↔ docs).
  • Results come home. Preview links, screenshots, and test outcomes flow back to Slack & Linear so non-devs never have to open GitHub.

Open decisions (marked ❓ on the diagram)

  1. Slack intake: when someone asks a question in Slack, does it become a GitHub issue, a Linear issue, or a Notion doc — and who/what decides which?
  2. Source of truth: is GitHub authoritative (agents need this) with Linear as a mirror, or do humans treat Linear as the real board?
  3. Design-system loop: what is the actual Storybook↔Figma sync mechanism (e.g. Code Connect, a tokens pipeline)? Where does "propose to team" happen — a PR, or in Figma?
  4. HubSpot: do marketing pages get built by agents (as code → deployed) or authored directly in HubSpot, with GitHub just tracking the initiative?
  5. Mac mini: which Apple things — app builds, signing, physical device tests? This decides what the runner must be wired to.
