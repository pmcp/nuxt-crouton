Schema review — Recursive nested layouts WS2 · keystone

Data-model sign-off for #871 (epic #868). Schema sign-off gate (#314) — agree the persisted shape before any code lands.

What changes

Today a layout leaf hosts exactly one block. WS2 lets a pane be a whole sub-layout — so a page is a layout whose panes are app-layouts, recursively ("layouts in layouts").

✅ Recommended — a dedicated nested node

leaf stays "one block"; split stays "arrange siblings in this layout"; the new nested means "an app boundary that is itself a layout."

interface LayoutNested extends LayoutNodeBase {
  type: 'nested'
  layout: LayoutTree   // a full sub-layout (own renderer +, later, own breakpoints)
  label?: string       // app name shown at the App zoom level (WS1)
}
type LayoutNode = LayoutLeaf | LayoutSplit | LayoutNested
FieldTypeReqNotes
type'nested'✓discriminator (joins 'leaf'/'split')
layoutLayoutTree✓the embedded sub-layout — an "app" that is itself a layout. Full tree so it can carry its own renderer (#855) + breakpoints (WS5)
labelstring—app name at the App zoom level (WS1); falls back to the primary block name
defaultSizenumber 0–100—size within the parent split (from LayoutNodeBase)
minSizenumber 0–100—min size within the parent split (from LayoutNodeBase)

LayoutLeaf and LayoutSplit are unchanged. Only the union widens; one node type is added. No DB migration — layout_configs.tree is already a JSON column.

Decisions needing sign-off

  • Full LayoutTree vs bare root in layout — recommend full (forward-compat with #855 + WS5).
  • Depth cap — nesting must add to the existing MAX_DEPTH=12 guard; or give nesting its own cap (~3, matching Site→Page→App)?
  • Edit path space — recommend each sub-layout edits in its own NodePath space, scoped to the focused zoom level.

Considered & rejected

  • A · overload the leaf (blockId? + layout?) — the brief's literal wording. ❌ as the mechanism: makes blockId optional → every consumer branches "block or layout?"; a dedicated node keeps leaf honest.
  • C · reference by id (layoutRef → another row) — ❌ deferred: enables reuse + composes with the site flow, but adds resolution/fetch/cycle-detection. Inline embedding can gain refs later.
  • re-use split as the boundary — ❌ no way to say "these panes are one app" → kills the App zoom level + per-app breakpoints.
