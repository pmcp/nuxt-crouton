Part II · the feedback loops
Post → command → auto → act → re-post
Each loop: 🟡 the agent posts an artifact · 🟢 you command on it · 🟣 an automatic comment fires · 🟠/🔵 a skill or workflow acts · ✅ it ends on your lgtm.
A · Annotate a live preview — the UI gate (#307/#488/#590/#607)
🟡 Agent posts a live staging preview. The cf:staging build with NUXT_PUBLIC_CROUTON_REVIEW=true stamps every component root with data-crouton-src, ships the Annotate tool + the /api/_review bridge, and drops a sticky PR comment with the URL.
🟢 You command on the page — open the 🥽 glasses launcher (bottom-right) → toggle Annotate → hover highlights elements → click one → type → Send. (Static-mockup fallback: inline-comment the committed .md.)
🟣 Auto-comment fires. useCroutonAnnotate builds a ReviewAnnotation → POST /api/_review posts as crouton[bot] (GitHub-App token, no PAT) the exact comment the agent keys off:
🎯 Preview feedback - Component: `app/components/ProductCard.vue` - Element: `div > button:nth-of-type(2)` (bbox 100,200 150×40) - Page: `/products/view` > make this smaller and add an icon
🔵 Agent acts. The subscribed agent (subscribe_pr_activity) wakes on 🎯 Preview feedback, reads the Component line, edits that file, redeploys to the same URL, appends - rN: … to the sticky.
✅ Ends when you reply lgtm / approve. No PR open? → the bridge opens an issue instead (#812).
B · Edit a ticket diagram — round-trip (#479/#483)
🟡 Agent posts 👤 /ticket-diagram → an Excalidraw status PNG — boxes coloured by status, dependency arrows, with the editable scene embedded inside the PNG (a tEXt chunk). Committed <slug>.png + diffable <slug>.graph.json, sticky comment. Renders on the GitHub mobile app.
🟢 You command on the diagram — open the PNG in Excalidraw (the scene loads live) → drag boxes / edit labels → export with “Embed scene” ON → attach the edited PNG back to a comment. (Mobile: one-tap ticket-editor Worker — each Save posts a frozen comment.)
🟣 Round-trip back in. 🛠 ticket-excalidraw-import.mjs runs extractScene() to decode the scene from your PNG → re-renders from your layout → embedScene() re-embeds it → commits the refreshed PNG, appends the revision log.
✅ Ends on lgtm / approve → the gate merges.
C · Sign off the data model — schema gate (#314)
🟡 Agent posts 🛠 schema-review → a field-table PNG + commits <collection>.md (the inline-commentable surface) and .html.
🟢 You command on the table — inline-comment the .md in the PR diff, pinned to a field: “make this decimal”, “add a slug field”.
🔵 Agent acts. Reads the review threads (pull_request_read), edits schemas/<collection>.json, re-renders, replies/resolves each thread, pushes — the diff updates in place.
✅ Ends on lgtm / approve → only then does crouton config generate code.
D · Agree on the test — test gate (#774)
🟡 Agent posts 🛠 test-review → the proposed failing test — a plain-language case list + the test code, committed red. (Logic in packages/* only.)
🟢 You command on the cases — inline-comment the test (“add a case for …”, “fix this expectation”) or just say so in chat.
🔵 Agent acts. Revises cases/expectations, replies to threads — the test stays in the diff as the contract.
✅ Ends on lgtm / approve → only then writes the code to make it green.
⚑ The one signal that resumes any held gate (#572/#310)
✅ Only an issue comment matching /\b(approve|lgtm)\b/i on a status:blocked issue. ⚙️ resume-on-comment.yml then merges the gate PR with an App token (so pull_request: closed cascades to wave advancement) — merge, not squash.
🚫 A 👍 reaction, the ui-approved label, or marking the draft ready do nothing — no workflow listens for them.
…and the loops with no human in them (machine ↔ machine)
🔧 Autofix — ⚙️ fix-ci-on-failure.yml on CI/E2E red → a bot reproduces & fixes, pushes via App token (re-triggers CI). Capped at 3 (ci-fix-attempts:N); never touches packages/; at the cap it 🛑 blocks the issue + tags you.
🔀 Auto-merge — ⚙️ automerge-epic-subpr.yml: a non-draft, pipeline-authored sub-PR onto an epic/ branch with no red checks and no packages/ changes merges itself.
🌊 Waves — ⚙️ schedule-waves.yml: each issue's Blocked-by: #NN edges; when all blockers close it applies delegate (App token) → the next worker dispatches.
💬 Comment dispatch — you comment 👤 /delegate or /deploy on an issue → ⚙️ comment-dispatch.yml applies the delegate label, reacts 🚀, the pipeline picks it up.
📊 Status note — ⚙️ pipeline-pr-status.yml: a sticky <!-- crouton-pipeline-status --> per PR + epic roll-up, refreshed on every check-suite.
🧪 Needs your eyes — 🛠 epic-digest's top band surfaces each merged PR's authored how-to-test steps + a 👁 badge for visual changes, so QA has a ready checklist.
The whole thing in one breath
idea → 🛠 github-tasks (issue first) → gates that each post an artifact 🟡, take your command 🟢, fire an auto-comment 🟣, and re-post until you reply lgtm → 🛠 crouton build + typecheck → 👤 /commit → PR → ⚙️ CI fans out (autofix / automerge run themselves) → merge = staging → 👤 /deploy-production → 🛠 postmortem.
Burn in: the issue exists before code, and an lgtm/approve comment — not a reaction, not a label — is the only thing that unblocks a held gate.
