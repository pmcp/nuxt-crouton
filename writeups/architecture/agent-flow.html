From your ticket to shipped code — the road

You write a ticket; the agents take it from there — fanning it out, pausing at gates where only you can decide, and handing the result back as something you can see and click. This is that road, lane by lane: what you do, what the agents do, and the skill that fires at each stop.

Highlighted = a moment you are in the loop. Everything else the agents run on their own.

▸ Zoom: the recursive fan-out

Steps 2–4 above, drawn out. One task becomes a tree — each node decides split or build, and workers run as isolated parallel leaves.

Every node runs the LEAF TEST. Orchestrator and decomposer only read/write issues, so they run on Sonnet; only the worker writes real code, on Opus.

💬 Comments are the steering wheel

There's no dashboard and no buttons — the whole thing is driven by comments and labels on issues and PRs. That's how you direct it, and how it talks back. Each of these is a real GitHub Action listening for a real comment.

You → @claude on any issue or PR

Type @claude with what you want; an agent picks up that comment and acts on it.

claude.yml · issue_comment / review_comment

You → reply on a blocked issue

Any comment on a status:blocked issue wakes the paused agent and it continues — no command needed.

resume-on-comment.yml

You → label an issue auto-decompose

Open or tag an issue with auto-decompose and the whole fan-out pipeline kicks off on its own.

decompose-on-issue.yml · opened / labeled

You → inline-comment a draft PR

Comment on the committed mockup or schema; the agent revises it in place, then resumes when you reply lgtm / 👍.

sign-off revision loop (#310)

Agent → comments back

The agent's own comments are the output: it asks for help, posts the screenshot, the preview URL, and the CI status.

add_issue_comment

↻ It keeps flowing — work creates work

The road isn't a dead end. A finished task routinely mints the next one — so the pipeline loops instead of stopping.

New issues & epics get minted four ways — recursion (a decomposer opens sub-issues), a request (your @claude or auto-decompose comment spins up a task or a whole epic), a regression (a post-merge break opens a fresh issue, #336), and a follow-up (the verify or review pass surfaces more). Each drops a new issue on the board — and the agents pick it up again.

⛨ What keeps the road safe

The pipeline can run unattended, so the guardrails are structural, not vibes.

epic branch siblings integrate before main — no duplicate scaffolding wave-gating foundation issues run before the work that depends on them packages/ HARD GATE a hook blocks shared-code edits unless the epic is approved no-squash atomic, blameable commits land on main — built for later archaeology block, don't improvise a missing prerequisite is a STOP, never a guess
