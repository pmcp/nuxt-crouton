3 The mental model

The parts that trip people up.

Preview is Staging

Same Workers env, same NAME.pmcp.dev domain, same NAME-staging-db. They differ only by trigger: a PR previews that PR; a push to the staging branch refreshes the shared env. It's one long-lived environment, not a per-PR throwaway.

Production is manual-only

workflow_dispatch with env=production → cf:deploy → NAME.friendlyinter.net. No seed step ever runs against prod.

First deploy auto-provisions

An id-less wrangler.jsonc lets the first deploy create D1 + KV; sync:ids writes the real ids back. No manual resource juggling.

The seed lives in the app script

cf:staging ends with db:seed:staging — not CI. That's why adding --with-staff makes the admin login appear on staging.

4 Per-app domains & preview scope

Concrete domains today, and one real nuance: the PR-preview path filter differs per app.

AppProductionStaging / PreviewPR preview triggers on
velo velo.friendlyinter.net velo.pmcp.dev apps/velo/** only
triage triage.friendlyinter.net triage.pmcp.dev apps/triage/**
fanfare kassa.pmcp.dev legacy naming kassa-preview.pmcp.dev apps/** + packages/**

Why a package change previews on fanfare but not velo

fanfare's pull_request trigger lists packages/** in its path filter, so a shared-package change (e.g. a crouton-core UI fix) auto-deploys a fanfare preview on the PR. velo's pull_request trigger only watches apps/velo/** — package changes deploy velo only on a push to the staging branch, never on the PR itself.
