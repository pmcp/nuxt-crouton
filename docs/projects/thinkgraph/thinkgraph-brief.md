# ThinkGraph — Vision Brief

---

## The core model

ThinkGraph is **not a chat interface that happens to be spatial.**
It is a **coordination layer between humans and AI**, structured as a graph of discrete units of work.

The fundamental primitive is the **node** — not the message, not the conversation.
A conversation is bound to a node. It is the thinking that happens *inside* that unit of work.
It is hidden by default, expandable on demand. It is never the thing you zoom out and see.

What you zoom out and see is **work flowing**.

---

## What a node is

Every node is a discrete unit of work with four parts:

| Part | Description |
|---|---|
| **Brief** | What needs to happen here. The input. Written by a human, generated by AI, or inherited from a parent handoff. |
| **Conversation** | The thinking, discussion, or AI session bound to this node. Collapsed by default. |
| **Output** | The artifact or conclusion. A decision, a file, an image, a summary, a plan. |
| **Handoff** | What happens next. |

A node is **done** when it has an output and a handoff. Not when the conversation stops.

---

## What a handoff is

A handoff is the mechanism that turns a completed node into the next unit of work.
It is explicit, not implied. Every node must declare one.

Handoff types:

| Type | What happens |
|---|---|
| **Task** | Output becomes a task brief. Written to Notion. Waits to be executed. |
| **Claude Code session** | Output becomes the brief for a new Claude Code session in a new git worktree. |
| **Human review** | Output goes to a named human or role. Node blocks until they respond. |
| **Child nodes** | More thinking is needed. Output seeds one or more child node briefs. |
| **Send** | Output is dispatched to an external AI (Claude, GPT, Flux, etc.) with a chosen execution mode. |
| **Close** | Work is done. No further action. Node becomes a permanent artifact. |

The handoff is what makes the graph a **coordination layer** rather than a chat log.
Without explicit handoffs, nodes accumulate. With them, work moves.

---

## The status lifecycle

Node states reflect the lifecycle of a unit of work, not the state of a chat:

| Status | Meaning |
|---|---|
| **Draft** | Brief exists, not yet started. Proposed by AI, awaiting human approval. |
| **Idle** | Approved, waiting to be picked up. |
| **Thinking** | AI or human is reasoning inside the node. |
| **Working** | Active execution — Claude Code session, Send in progress, human working. |
| **Needs attention** | Blocked. Waiting for input, decision, or review. |
| **Done** | Output exists, handoff declared. |
| **Error** | Something failed. Needs diagnosis. |

---

## Auto-summarisation

When a node's conversation concludes, it is automatically summarised into:
- A **brief** — what was decided or produced (the output)
- A **handoff** — what happens next

This is the mechanism that turns conversation into structure. The AI does not just chat —
it concludes. The summary becomes the artifact that travels downstream, not the raw conversation.

For Claude Code sessions: the natural "handoff brief" Claude Code already generates at session
close is exactly this artifact. Small, structured, portable. It becomes the node output.

## Data hierarchy

```
Notion Epic
  └── User Story (Notion sync) → canvas path root
        └── Tasks (AI-proposed → human-approved → Notion)
              └── Claude Code session nodes (execution)
                    └── Generated nodes: decisions, questions, milestones
                          └── Human nodes: forks, remarks, reviews
```

**Notion** is the authority for planned work (epics, user stories, tasks, kanban).
**ThinkGraph** is the authority for execution (sessions, decisions, context, handoffs).

ThinkGraph writes back to Notion on two events only:
- Task approved → `create_task`
- Task node done → `update_status`

---

## The planning pass

When a User Story node lands on the canvas, ThinkGraph does a **planning pass** before execution:

1. Reads the user story brief + epic context
2. Reads sibling story nodes for cross-context
3. Proposes a task breakdown as **draft** child nodes (dashed border, no step number)
4. Human reviews, reshapes, approves
5. Approved nodes solidify → written to Notion → armed for execution

This replaces the planning meeting. The AI proposes, the human decides.

---

## Context model

Each node inherits context from its ancestors via their **output briefs** — not raw conversation.
Context scope is configurable per node:

- **Full** — entire ancestor chain of briefs
- **Branch** — direct parent chain only
- **Manual** — cherry-pick any nodes from anywhere on the canvas

Because context travels as briefs (not raw output), token usage stays bounded regardless
of tree depth. Deep chains can be compressed further by an explicit summarisation node.

---

## Collaboration (Yjs)

The canvas is a Yjs document. Real-time multi-user by default.
Presence is visible — teammates appear hovering nodes before they act.

Human participation modes per node:
- **Direct** — human input feeds immediately into the active session
- **Queue** — input is held until the session owner approves injection
- **Fork** — always starts a new branch, never touches the parent session

---

## Visual language

The canvas communicates work state without opening anything:

| Status | Visual |
|---|---|
| Thinking | Slow pulse |
| Working | Progress ring + animated bar |
| Needs attention | Orange glow, subtle shake |
| Done | Settled, solid |
| Draft | Dashed border, reduced opacity |
| Error | Red, urgent pulse |

When a node is selected, its context chain highlights — ancestor nodes illuminate,
edges animate in the direction of data flow, non-context nodes dim.
You can see what a node knows without opening it.

---

## Scale strategy

- **One canvas per epic** — scope-bounded, closes when epic is done
- Closed canvases are permanent artifacts: the full execution history of that epic
- User story paths collapse/expand — the canvas never becomes a blob
- Epic overview: read-only progress panel per user story (done / working / blocked)
  derived from node states, no extra sync

---

## What this is not

- Not a chat interface. Conversations are inside nodes, not the canvas itself.
- Not a kanban. Notion owns the board. ThinkGraph owns the execution.
- Not a terminal replacement. Claude Code runs in worktrees. ThinkGraph coordinates it.
- Not a solo AI tool. It is designed for humans and AI working together, with clear
  handoffs between them, visible to the whole team in real time.

## Open questions

1. **Handoff approval** — which handoff types require human confirmation before firing?
2. **Context window depth** — when to auto-trigger summarisation on deep ancestor chains
3. **Skill versioning** — which version ran on a node affects resume/remix behaviour
4. **Worktree lifecycle** — close/merge/abandon state needed to trigger cleanup
5. **Cost visibility** — token counter per node and per run
6. **Emergent tasks** — AI-discovered subtasks born on canvas: when/how to push back to Notion
7. **Simultaneous forks** — two teammates fork same node at once: both run? queue? conflict UI?
